## RVO и NRVO. Move semantics. Move конструктор & Move Assignment operator. Голяма 6-ца (Rule of Five)

### RVO, NRVO 

**(Named) Return Value Optimization** е оптимизация, предоставена от повечето съвременни C++ компилатори. Тя цели избягването на излишни копирания (извикването на копиращи конструктори) на временни обекти (например върнати по копие от функция).

**Пример:**

```c++
class A
{
private:
    int a;
    
public:
    A() : a(0)
    {}
    
    A(int a) : a(a)
    {}
};


A makeObj()
{
    return A(5);
}

int main()
{   
    A obj1 = makeObj(); // Заради RVO ще се извика само конструктора с един параметър на временния обект,
                        // върнат от функцията. Няма да се извърши копиране.
    
    А obj2 = A(2); // Отново RVO, избягва се копиране на временен обект
    
    А obj3; // default-ен конструктор
    Obj3 = makeObj(); // Тук няма RVO, първо се създава временния обект от функцията, след това
                     // се присвоява от вече съществуващия obj3 с оператор =
    
    return 0;
}
```

Примерът за **NRVO** е аналогичен, но с разликата, че върнатия от функцията обект се създава по-рано в нейния scope.
```c++
A makeObj()
{
    A obj(5);
    return obj;
}
```



### Типове стойности в С++

### lvalue & rvalue
Всеки израз (**expression**) в C++ е **lvalue** или **rvalue**.
- **lvalue** - Израз, обозначаващ обект или функция, с конкретно място в паметта, т.е. нещо, което "живее" в паметта и продължава да съществува след текущия ред. (e.g. именована променлива, функция и др.)

- **rvalue** - Израз, който не е *lvalue*. *Rvalue* изразите се разделят на два вида:
    - **prvalue** (pure value) - Израз, без конкретно място в паметта, т.е. "чиста" стойност, която не се отнася до съществуващ обект. (e.g. `42`, `3.14`, `x + 1`, `'a'`, `5 * 5`)
    - **xvalue** (eXpiring value) - Израз, обозначаващ временен обект с конкретно място в паметта, който скоро ще бъде изтрит.

#### Примери:
```c++
int num;
num = 4;       // Okay
```
Операторът `=` изисква (modifiable) *lvalue* от лявата си страна, което в случая е `num`.

```c++
4 = var;        // Error
(var + 1) = 4;  // Error
```
В случая нито `4`, нито `(var + 1)` са *lvalue*, затова компилаторът дава грешка.

---

*lvalue*-тата могат и да не са променливи:
```c++
int x;

int& getRef() 
{
	return x;
}

int main() 
{
	getRef() = 4;  // Okay, getRef() is an lvalue - returns a reference to the global variable x
	return 0;
}
```
Тук `getRef()` връща референция към глобалната променлива `x`, която има адрес в паметта и е *lvalue*, т.е. всичко е наред. <br />

Колкото при *rvalue*-тата:
```c++
int x;

int getRef() 
{
	return x;
}

int main() 
{
	getRef() = 4;  // Error
}
```
Тук `getRef()` вече е *rvalue* - вместо да се връща референция към обекта `x`, се връща някакво негово локално копие.

### rvalue reference
В следните примери използваме int за простота. На практика, както вече знаем, няма смисъл да подаваме int по референция, защото копирането е еднакво евтино.

До C++11 референции към константна *rvalue* са възможни, но те не могат да изменят обекта:
```c++
int val = 42;

// lvalue references
int& ref = val;             // Okay - lvalue ref може към lvalue
int& ref1 = 20;             // Error - lvalue ref не може към rvalue
int& ref2 = val * 2;        // Error - lvalue ref не може към rvalue
const int& cref1 = 20;      // Okay - const lvalue ref може към rvalue
const int& cref2 = val * 2; // Okay - const lvalue ref може към rvalue
```

В C++11 се въвежда нов тип референция, а именно *rvalue* референцията, която позволява референция към *rvalue*. С други думи, *rvalue* референциите са идеалният начин за разпознаване дали един обект е временен или не. За разлика от *lvalue* референциите, те се пишат с &&. <br />
```c++
int&& rref1 = 20;           // Ok - rvalue ref към rvalue
int&& rref2 = val * 2;      // Ok - rvalue ref към rvalue
int&& rref3 = val;          // Error - rvalue ref не може към lvalue
```

Въпреки, че *rvalue* референции не могат да сочат към *lvalue*-та, можем да cast-нем *lvalue* към *xvalue* (в съответстващата ѝ rvalue референция)
```c++
int&& rref3 = std::move(val);
```

На практика `std::move` казва на компилатора да третира `x` като *rvalue*. (конвентира `x` ,който е *lvalue*, към *xvalue*) <br />

Добре, с какво ни помагат този вид референции? <br />
Ами, вече разполагаме с механизъм, който еднозначно ни казва кога един израз е *lvalue* и кога е *rvalue*, т.е. можем да предефинираме една функционалност да работи по два различни начина - по един при работа с *lvalue* изрази, а по друг с *rvalue* такива.

```c++
void f(int& x) 
{
    std::cout << "Called with an lvalue: " << x;
}
 
void f(int&& x) 
{
    std::cout << "Called with an rvalue (xvalue or prvalue): " << x;
}

int main() {
	int num = 20;
	f(num);  			// Takes an lvalue => calls f(int&)
	f(10);  			// Takes an rvalue as a parameter => calls f(int&&)
	f(std::move(num));  // Takes an rvalue as a parameter => calls f(int&&)
}
```

А защо е толкова съществено да разполагаме с начин, по който да разграничаваме кога един израз е *lvalue* и кога е *rvalue*?
Нека си представим, че разработваме приложение за слушане на музика като Spotify и имаме два класа:
- `class Song`, който моделира една песен и е с размер >= 64 байта (смята се за голям по памет клас).
- `class Playlist`, който моделира плейлист с динамичен масив от песни.

Нека си представим, че имаме инстанция `Playlist A` с 1 милион песни, която скоро ще бъде изтрита и знаем, че повече не ни трябва. Преди това искаме да създадем `Playlist B` със същите данни като `Playlist A`. С досегашните ни знания единственият вариант е да копираме `Playlist A` в `Playlist B`, което ще създаде нов масив от 1 милион песни и ще ги копира една по една. След което при изтриването на `Playlist A` ще се изтрият старите 1 милион песни. Както знаем, копирането и изтриването са много бавни операции, особено при големи по памет класове.

C++11 ни предоставя възможност да решим този проблем чрез въвеждането на Move конструктор и Move assignment оператор.

### Move конструктор & Move Assignment оператор

Move конструкторът, приема референция към същия тип, но за разлика от copy конструктора, я приема по неконстантна rvalue референция. <br />
Идеята е следната - при създаване на обект, крадем ресурсите на обекта, към който е референцията и го оставяме в такова състояние, че последващото му изтриване да не навреди на новосъздадения обект. <br /> 
При примитивните типове данни това е просто копиране. Вече при указателите, при които наша имплементация на copy контрола би наложила дълбоко копиране, е малко по-различно. <br />
```c++
Playlist::Playlist(Playlist&& other) 
{
    length = other.length;
    songs = other.songs;

	other.songs = nullptr;   // Leaves toMove in a state that further destruction will do no harm!
}
```
Т.е. слагаме songs да сочи към парчето памет, към което сочи toMove.songs (а не заделяме наново динамична памет!), също казваме length = toMove.length. <br />
В тялото слагаме името на toMove да сочи към nullptr, т.е. при изтриване той няма да навреди по никакъв начин на обекта, в който местим ресурсите. <br />

```c++
Playlist& operator=(Playlist&& other)
{
	if (this != &other) 
	{
		free();	// Free the resources associated with this

		length = other.length;
		songs = other.songs;

        other.songs = nullptr;  // put other in a state where deleting won`t cause problems in further uses of this
	}

	return *this;
}
```

### Синтезирани move операции
Както и при copy конструктура, copy assignment оператора и деструктура, компилаторът автоматично ще синтезира move конструктор и move assignment оператор. <br />
Единствената разлика тук е, че условията, при които той прави това са различни. <br />
За разлика от copy операциите, за някои класове компилаторът изобщо не създава move конструктор и move assignment оператор. <br />
В частност, ако един клас дефинира свой copy конструктор, copy assignment оператор или деструктор, то move конструкторът и move assignment операторът няма да бъдат синтезирани. Aко един клас няма move операции, то се използват copy такива (даже върху временни обекти, като точно това целим да избегнем), ако такива има разбира се. <br />
Компилаторът ще създаде move конструктор и move assignment оператор ако класът няма дефиниран copy control и ако всеки член може да бъде "преместен". <br />
Компилаторът може да премества вградени типове, а също и класове, които имат съответната move операция дефинирана. При примитивните типове данни местенето е просто копиране. <br />
Aко експлицитно помолим компилатора да генерира move операция чрез =default, и компилаторът не успее да синтезира такъв, то той бива маркиран като =delete.
